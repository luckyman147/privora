## Problem Statement

Existing form platforms (Google Forms, Typeform, Microsoft Forms) have fake anonymity: they log IP addresses, link sessions to Google accounts, and give organizational administrators full audit trail access. Users who need genuinely anonymous data collection — for elections, sensitive feedback, whistleblowing, course evaluations — either use these tools and accept the privacy theater, or avoid collecting data altogether.

The specific gap: no self-serve, affordable tool exists for small organizations (student clubs, community groups, NGOs) that need to run trustworthy elections or anonymous surveys. Enterprise-grade solutions (SimplyVoting, ElectionRunner) exist but are priced and scoped for institutions, not individuals.

The novel angle: existing privacy tools make privacy a *configuration toggle*. SecureForm makes trust a *verifiable artifact visible to the submitter* — a live Trust Score displayed on every form showing what data is collected, who can access it, whether submissions are truly one-per-person, and how long data is retained.

## Demand Evidence

**Strongest signal:** "I've seen people avoid using forms altogether because of these concerns."

Avoidance behavior is stronger than feature complaints. When users abandon a category entirely rather than use an imperfect tool, the unmet need is acute enough to represent a real market. SecureForm isn't competing with Google Forms — it's enabling data collection that currently doesn't happen.

**Documented pain (from landscape research):**
- Google Forms logs IP addresses on every submission, even with "anonymous" settings enabled
- Session linking: signed-in Google users can be identified on the backend even without explicit sign-in
- Google Workspace admins have full audit trail access — "anonymous" is not enforceable in org contexts
- The question "is Google Forms really anonymous?" is one of the most searched in this category — real, widespread anxiety

**Current validation stage:** Hypothesis. No specific named customers yet. The assignment section addresses this directly.

## Status Quo

Users running elections or collecting sensitive feedback today use Google Forms because it's free and accessible. The actual workarounds:
- Run a paper vote and count manually (time-consuming, not scalable)
- Use Google Forms and just hope people trust it (low trust = low participation)
- Skip collecting data entirely (the avoidance behavior above)
- Pay for SimplyVoting or ElectionRunner at ~$2-5/voter (too expensive for a 30-person student club)

The cost of the status quo: decisions get made with less information, elections lose legitimacy, sensitive feedback goes uncollected.

## Target User & Narrowest Wedge

**Primary target:** Student association president running an election.

- What they need: a tool that enforces exactly one vote per member, keeps votes anonymous, and makes that anonymity verifiable (not just claimed)
- What's at stake: their reputation. If the election result gets challenged because the tool was gamed or leaked voter identity, their leadership legitimacy collapses
- What they currently use: Google Forms, WhatsApp polls, or paper
- What they'd pay: unclear — probably not much in the beginning, but they'd use and recommend a free tool that works
- Why they're the right wedge: concentrated pain, clear success criteria (did the election run and get trusted?), easy to reach (every university has 5-20 of these people)

**Distribution unit (revised):** The individual club president with reputation skin in the game — NOT the IT department or institution. Universities move slowly; clubs dissolve annually. The person who needs it is the person who adopts it.

## Constraints

- University final-year project (PFE) context — needs academic deliverability by a specific deadline
- Solo developer (or small team) — complexity must stay manageable
- No budget for enterprise infrastructure at launch
- Target geography: Tunisia initially, then Arabic/French-speaking markets and beyond
- Startup aspirations beyond the academic deliverable

## Premises

1. The real MVP customer is student associations and small orgs running elections/votes, not enterprises or HR departments.
2. The wedge is NOT "privacy-first forms" (that market is crowded: BlockSurvey, Hushwork, Zonka Feedback). The wedge is "verifiable, visible trust at the point of submission" — the Trust Score concept.
3. MVP scope: minimum viable form builder (5-6 question types) + one-time-submission guarantee + Trust Score display. Not AI question generation, not file upload, not digital signatures.
4. Distribution = individual club presidents with reputation at stake, not institutional channels.
5. AI question generation, conditional logic, matrix questions, digital signatures — all post-MVP. They don't solve the core trust problem.

## Approaches Considered

### Approach A: Election First — Minimum Viable Voting Tool
Build only what a student president needs to run a trustworthy election: candidate list, single-choice vote, email-based one-vote enforcement, static Trust Score badge.

- Effort: S (2-3 weeks solo / ~2-3 days with CC+gstack)
- Risk: Low
- Pros: Laser-focused on the wedge customer; ships fastest; email verification is stronger than localStorage token for one-person-one-vote
- Cons: "Voting tool" framing limits growth path to surveys and feedback; harder to expand scope after launch
- Reuses: Supabase auth for creator accounts; email OTP for voter identity

### Approach B: Platform MVP — Form Builder + Trust Architecture *(chosen)*
Next.js + Supabase, 5-6 question types (short text, long text, multiple choice, checkbox, rating, dropdown), token-based one-time submission enforcement, dynamic Trust Score that reflects actual form configuration.

- Effort: M (6-10 weeks solo / 2-3 weeks with CC+gstack)
- Risk: Medium
- Pros: Broader use cases from day one (elections, surveys, anonymous feedback, course evals); platform positioning enables expansion; dynamic Trust Score is more credible than static claims
- Cons: Longer to ship — risk of building before validating the wedge; more surface area to maintain
- Reuses: Supabase RLS for data access control; JWT for creator authentication

### Approach C: Trust Layer — The Badge, Not the Form
Build the Trust Score as an embeddable badge + verification API that augments any form platform. Creators configure on Google Forms or Typeform, add the Trust Score badge. You own the standard, not the form.

- Effort: M (4-6 weeks / 1-2 weeks with CC+gstack)
- Risk: High
- Pros: Enormous moat if the standard gets adopted; no direct competition with incumbents; the SSL padlock analogy holds perfectly
- Cons: Acquisition requires integration partners, not direct users; doesn't solve one-time-submission at the form level; harder to find first users

## Recommended Approach

**Approach B: Platform MVP**

User chose B (broader use cases from day one). The strategic bet: a minimum form builder with 5-6 question types gives the platform enough surface area to serve elections, surveys, and anonymous feedback from launch. The dynamic Trust Score (reflecting actual configuration rather than static claims) is more defensible against copycat features.

**Implementation priority order:**
1. Creator authentication (Supabase auth, email/password + magic link)
2. Form builder with 8 question types (see full list below)
3. One-time submission enforcement (see spec below)
4. Trust Score display (see spec below)
5. Results dashboard (see spec below)

Note: Creator auth comes first because election forms require email-based voter verification, and auth infrastructure is shared. Building one-time submission before auth creates a dependency gap.

**Stack recommendation:** React (Vite) + Supabase (auth, DB, RLS, Edge Functions) + Vercel deployment.

- Frontend: React + Vite (SPA, client-side routing via React Router, no SSR)
- Backend/API: Supabase Edge Functions (Deno) for server-side logic — submission token generation, hash validation, email dispatch via Resend, cron cleanup trigger
- Auth: Supabase Auth (email/password + magic link)
- Database: Supabase PostgreSQL + Row Level Security
- Hosting: Vercel (static React build) + Supabase (DB + Edge Functions)
- Cron: Vercel Cron Jobs call a Supabase Edge Function nightly for retention enforcement

**One-time submission specification:**

Two tiers based on form type (creator-selected at form creation):

*Anonymous survey mode (default):*
- Primary enforcement: UUID token generated server-side at first page load, stored in `submissions` table hashed (SHA-256). Stored in localStorage on client.
- Secondary enforcement: browser fingerprint computed from `navigator.userAgent + screen.width + screen.height + Intl.DateTimeFormat().resolvedOptions().timeZone` — concatenated and hashed. Do NOT use FingerprintJS (external dependency, privacy contradiction). Use native browser APIs only.
- Fingerprint role: heuristic flagging, not hard block. If token present → block. If token absent but fingerprint matches existing submission → show warning ("looks like you may have already submitted") + require explicit "I haven't submitted before" confirmation before allowing. Fingerprint alone never blocks silently.
- Accept incognito bypass as known limitation — document it in Trust Score ("1 submission per browser session").

*Election mode (creator opt-in):*
- Requires email verification. Creator uploads an allowlist of eligible voter emails (CSV) or manually enters them.
- Voter receives a unique one-time link via email (UUID token, NOT a numeric OTP). Link format: `https://app.secureform.io/vote/[token]`.
- Once link is visited and form submitted, token is marked used. Link expires.
- No anonymous token needed — email identity IS the one-vote mechanism.
- Voter email is never stored alongside their vote (decoupled: `voter_tokens` table links token→email, `responses` table links token→answers, joined only at submission time then token marked used and decoupled).

*Duplicate submission error state (both modes):*
- Anonymous: "It looks like you've already submitted this form. If this is a mistake, contact the form creator." No retry option exposed by default.
- Election: "This voting link has already been used. Each link works once. If you believe this is an error, contact [creator email]." Hard block — no bypass.

**Trust Score specification:**

Trust Score badge displayed at the top of every form, before any questions. Computed from form configuration at render time — not static, not configurable via free text. Renders as a card with 4 rows:

| Row | Label | Possible values (determined by config) |
|-----|-------|----------------------------------------|
| Identity | Who can see who responded | "Nobody — responses are anonymous" / "Creator only — email-verified voters" |
| IP | Network data | "Your IP address is not stored in our database" *(always true — enforce this server-side; Vercel may log IPs in their infra, which is out of scope and disclosed)* |
| Submissions | Limit | "1 submission per browser session" (anonymous mode) / "1 submission per verified email" (election mode) |
| Retention | Data lifetime | "Responses deleted [N] days after form closes" (creator sets at creation; a scheduled Supabase function enforces it) |

Trust Score color: green (election mode with email verification) / blue (anonymous survey mode). Never shown as "fully private" unless election mode — anonymous token mode is weaker and labeled accurately.

The "IP not stored in our database" claim is the only technically enforceable IP claim. Do not promise "no IP logging" — Vercel's infrastructure logs IPs regardless of app behavior. The badge text must be precise: app-level, not infrastructure-level.

**Form builder question types (exhaustive list for MVP):**
1. Short text (single-line input)
2. Long text (multi-line textarea)
3. Multiple choice (radio buttons, single selection)
4. Checkboxes (multiple selection)
5. Rating (1-5 or 1-10 scale, creator configures range)
6. Dropdown (select element, single choice)
7. Matrix / comparison table (rows = items, columns = scale labels; each cell is a radio; useful for ranking candidates or comparing options)
8. Conditional question (show/hide a question based on the answer to a previous question; creator configures: if Question N answer = X → show Question M)

Note on complexity: matrix and conditional logic add significant form-builder UI work. Matrix is a fixed grid (manageable). Conditional logic requires a dependency graph stored in question_config jsonb — keep it simple: one condition per question (if/then, no AND/OR chains) for MVP.

No other question types in MVP. File upload, date/time, signature, multi-condition logic — deferred.

**Results dashboard specification:**
- Route: `/dashboard/forms/[formId]/responses`
- Columns displayed: submission timestamp, one column per question (question label as header, answer as value)
- Pagination: 50 rows per page, cursor-based (Supabase rangeFrom/rangeTo)
- Export: CSV download button — generates `responses-[formId]-[date].csv` client-side from fetched rows
- No charts, no aggregation, no sentiment analysis in MVP
- Election mode: CSV export includes answer columns only — no email column (emails decoupled from responses at submission time; not recoverable by design)
- Access: creator-only (Supabase RLS policy: `auth.uid() = forms.creator_id`)

**Data schemas (informal):**

```
forms: id, creator_id, title, mode (survey|election), question_config (jsonb), 
       retention_days, created_at, closes_at

questions: id, form_id, type, label, options (jsonb), position

responses: id, form_id, submission_token (hashed), submitted_at

response_answers: id, response_id, question_id, answer_value

voter_tokens: id, form_id, email_hash (SHA-256 of email), token (UUID), 
              used (boolean), used_at — NOTE: email_hash is one-way; 
              original email stored separately in voter_invites table 
              (used only for sending invite email, deleted after send)

voter_invites: id, form_id, email (plaintext), token_id — deleted after 
               email sent; no join possible post-send
```

**Email provider:** Resend (resend.com). Free tier: 100 emails/day, 3,000/month. Next.js SDK available (`npm install resend`). No SMTP config. Used for: election invite emails (one-time voter links), results notification to creator (optional). Do not use Supabase built-in email for this — rate limits are too low and template control is limited.

**Retention enforcement (free-tier compatible):** Do NOT use Supabase pg_cron (paid plans only). Use Vercel Cron Jobs (free on Hobby plan, runs on schedule via `vercel.json`). A cron route at `/api/cron/cleanup` runs nightly, queries forms where `closes_at + retention_days < now()`, and deletes associated `responses`, `response_answers`, and `voter_invites` rows. Vercel cron max frequency: every hour on free tier. Nightly is sufficient.

## Success Criteria

**PFE validation (academic):**
- Functional form builder with 5+ question types
- One-time submission demonstrably enforced (show test with attempted duplicates)
- Trust Score displayed on all forms
- At least 2-3 real forms run with real users (not just demo)

**Startup validation (market):**
- 5 club presidents run a real election or survey using the tool
- At least one person refers it to another club without being asked
- No reported trust incidents (no anonymity breach complaints, no credible reports of bypassed duplicate submission enforcement)
- Election mode: zero successful duplicate votes in post-election log audit (verifiable via `voter_tokens` table — all tokens used exactly once)

**The metric that matters most:** Can a student president send the link to 20 people, collect votes, and feel confident the result is legitimate? That's the product.

## Distribution Plan

Web service — no download required for submitters. Creator signs up, builds form, then distributes via two channels (both available in MVP):

**Channel 1: Shareable link** — Creator copies a public URL (`https://app.secureform.io/f/[formSlug]`) and shares via WhatsApp, Telegram, email, or any channel they choose. The platform generates this link; distribution is external.

**Channel 2: Send by email from the platform** — Creator enters recipient email addresses (comma-separated or one per line), clicks "Send". The platform dispatches individual emails via Resend containing the form link + a short Trust Score summary. In survey mode, all recipients get the same public link. In election mode, each email contains a unique one-time voter link (see One-time submission spec above).

Route: `/dashboard/forms/[formId]/distribute`
Fields: textarea for email addresses, subject line (pre-filled: "[Creator name] has shared a form with you"), optional personal message (1-2 sentences). No per-recipient customization in MVP.
Rate limit: 100 emails/send (Resend free tier max). Show error if over limit: "You can send to up to 100 recipients at once. Split into batches."

Deployment: Vercel (free tier for MVP). Supabase (free tier for MVP).

First distribution channel (for your own validation): walk into your university's student union, find 3-5 club presidents, show them a live demo. If they use it without being asked a second time, you have something. If they don't, diagnose why before building more.

## DevSecOps

Security and operations plan for a platform making explicit privacy guarantees. 
The Trust Score is a public contract — a breach here is a reputational catastrophe, 
not just a bug.

### CI/CD Pipeline (GitHub Actions)

```yaml
# .github/workflows/ci.yml — triggers on PR and push to main
jobs:
  test:
    - npm run lint          # ESLint (catch XSS patterns, unsafe practices)
    - npm run typecheck     # TypeScript strict mode
    - npm run test          # Vitest unit tests (submission logic, token hashing)
  security:
    - npm audit --audit-level=high   # dependency vulnerability scan
    - npx snyk test                  # deeper CVE scan (free tier available)
  build:
    - npm run build         # Vite production build (catches import errors)
  deploy:
    - vercel --prod         # auto-deploy on merge to main (Vercel CLI)
```

Supabase migrations run via `supabase db push` in CI — never applied manually 
in production. Migration files live in `supabase/migrations/` and are versioned 
in git.

### OWASP Top 10 — Relevant Mitigations

| Threat | Risk for SecureForm | Mitigation |
|--------|---------------------|------------|
| Injection | Form answer values inserted into DB | Supabase JS SDK uses parameterized queries by default. Never concatenate user input into raw SQL. |
| Broken Auth | Creator accesses another creator's forms | Supabase RLS: `auth.uid() = forms.creator_id` on all form/response reads. Tested in CI with RLS test suite. |
| XSS | Question labels rendered in form UI | React escapes all JSX output. Never use `dangerouslySetInnerHTML`. Sanitize any markdown-rendered content with DOMPurify if rich text is added. |
| IDOR | Submitter guesses another form's response ID | Response IDs are UUIDs (not sequential integers). RLS prevents cross-form reads. |
| Sensitive data exposure | Voter emails exposed in responses table | Voter email decoupled by design: `voter_invites` deleted after send, `voter_tokens` stores email_hash only, `responses` has no email column. |
| Security misconfiguration | Supabase anon key exposed in client | Supabase anon key is safe to expose (it's public by design). All privileged operations use service role key server-side only (Edge Functions env var, never in client bundle). |
| Rate limiting | Submission flooding, token brute force | Supabase Edge Functions: rate limit submission endpoint at 10 req/min per IP via a simple Redis counter or Upstash (free tier). Token format is UUID v4 — 122 bits of entropy, not brute-forceable. |

### Dependency Hygiene

- Enable GitHub Dependabot (`dependabot.yml`) for weekly npm dependency updates
- Pin major versions in `package.json` — avoid `*` or `latest`
- `npm audit` runs in CI and fails the build on high/critical CVEs

### Incident Response (Trust Score breach)

If a submitter's identity is exposed or a duplicate vote gets through:
1. Take the affected form offline immediately (set `forms.status = 'suspended'`)
2. Notify the form creator with exact incident details within 1 hour
3. Log the incident to `incident_log` table (form_id, type, discovered_at, resolved_at)
4. Post-mortem: which Trust Score claim was violated, root cause, fix

The Trust Score is a public promise. A single unaddressed incident kills the product. Treat it like a P0.
